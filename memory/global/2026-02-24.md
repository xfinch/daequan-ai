## Development Work

### Next.js Dashboard — DEPLOYMENT HELL 🔥
**Status:** Eventually deployed (48+ commits later)

**The Battle:**
Railway deployment failed repeatedly with cascading issues:

1. **MongoDB in Client Bundle**
   - Mongoose was being bundled client-side causing "Module not found" errors
   - Fixed by using `server-only` package, `serverComponentsExternalPackages`, and lazy-loading

2. **Next.js 15 Breaking Changes**
   - `searchParams` is now a Promise — required `React.use()` to unwrap
   - `PageProps` types needed updating across multiple pages
   - `lean()` query results lost type inference — added explicit `any` annotations

3. **Edge Runtime vs Node Runtime**
   - Middleware can't use Mongoose (dynamic code eval not allowed in Edge)
   - Moved auth checks from middleware to page level
   - Created auth-server.ts to isolate server-only concerns

4. **Docker/Railway Caching Nightmare**
   - Old navbar.tsx with inline `'use server'` was cached in Docker layers
   - Multiple cache buster commits to force fresh builds
   - Switched to Dockerfile for more control over build process
   - Added `.dockerignore` and explicit COPY order

**Final Solution:**
- Server Components fetch auth server-side, pass to Client Components as props
- Navbar and AdminNav now receive `user` and `session` as props
- KanbanBoardWrapper handles client-side interactivity
- Separate `lib/actions.ts` for server actions

**Commits:** 48+ fix commits — one of the most intensive debugging days in recent memory.

## Lessons Learned

1. **Cache invalidation is hard** — Docker layers, Railway builds, and Next.js caches all needed busting
2. **Next.js 15 changed a lot** — the Promise-based searchParams caught us off guard
3. **Server/Client boundaries matter** — mixing concerns caused most of the bundling issues
4. **Props > Direct auth() calls** — passing auth data down is more reliable than components calling auth()
